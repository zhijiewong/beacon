# Beacon datasets

Two datasets feed the index. **They differ in trust level — read this before using either.**

## 1. `snapshots/<YYYY-MM-DD>.json` — price data (REAL)
Live per-token prices for ~310 priced models, collected from the OpenRouter
`/models` API and normalized to USD per 1M tokens (blended 3:1 input:output).
Generated by `python3 -m beacon.collector`. Each listing carries its
`source_url` and `source_type`. This is genuine observed data.

**Known limitation (methodology section 11):** these are *quoted list prices*,
not cleared transactions. Transaction-grounding is a Phase 2/3 goal.

## 2. `benchmarks.csv` — capability data (CALIBRATED, GPQA-Diamond)
Maps `model_id` → benchmark score, used to place models into capability tiers.

**Status: 18 flagship models calibrated on GPQA-Diamond** (`verified=yes`), spanning
25.9 (Llama 3.1 8B) → 94.1 (Gemini 3.1 Pro). Scores were sourced 2026-06-07 from
public leaderboards, preferring a **single harness (pricepertoken)** for cross-model
comparability and falling back to Artificial Analysis / llm-stats only where it
lacked a model; the `source_url` is recorded per row. Overlaps were cross-checked
(Gemini 3.1 Pro = 94.1 on all three sources; Qwen3.7 Max 92.3/92.4).

**Deliberately omitted:** MMLU — saturated above ~88% across all frontier models, so
it no longer differentiates (methodology section 6 treats it as legacy). GPQA-Diamond
is the primary discriminating benchmark.

### Caveats / still to do
- Scores are *quoted leaderboard results*, harness-dependent (e.g. DeepSeek R1 is
  70.8 on pricepertoken vs 71.5 in its own paper). Mixing sources adds minor
  inconsistency — preferring one harness mitigates it.
- Add AIME / SWE-bench-Verified columns for multi-domain tiers (optional).
- Formalize thresholds in `tiers.json` (currently provisional in `beacon/analyze.py`).
- **Phase-1 acceptance test** (reproduce Epoch/a16z LLMflation curves) needs a *time
  series* of snapshots; only one snapshot exists so far. It accrues as the collector
  runs daily.
